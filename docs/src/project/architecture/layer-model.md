# Layer Model

JANUS organises its kernel code into a strict layered hierarchy. Dependencies flow downward only — a module may depend on any module in a lower layer, but never on a module in the same layer or above it. The build system enforces these rules via CMake target dependencies.

## Key Invariants

The `_start` layer creates `kernel.elf` by linking `kmain` and all subsystem libraries together. It contains the entry point in assembly, the protocol header (Multiboot2 or Limine), and the linker script that defines the memory layout.

`kmain` is the sole module that may depend on subsystems. This constraint is enforced by CMake — no other target is permitted to add a subsystem as a link dependency. The effect is that all inter-subsystem coordination flows through `kmain`, which acts as the composition root.

Subsystems are strictly isolated from one another. `boot` cannot call into `drivers`, and `drivers` cannot call into `mm`. Each subsystem exposes a public header directory and keeps its internal implementation private. If two subsystems need to share information, the data flows through `kmain` via an explicit parameter (typically the `kernel_descriptor_t`).

Libraries in `kernel/lib/` may only depend on the global interface layer. They are freestanding utilities — `page_tables`, `mem_alloc`, formatted I/O — and carry no knowledge of the subsystems that consume them. Libraries are also isolated from one another: no library may depend on any other library. Each one is self-contained.

The global interface layer (`kernel/include/janus/`) contains headers that are genuinely cross-cutting: `types.h` (fixed-width integer aliases), `attributes.h` (compiler attributes), and `config.h` (build-time configuration generated by CMake). A header is promoted to this layer only when it is used by a wide majority of modules; single-subsystem headers remain in their owning module.
